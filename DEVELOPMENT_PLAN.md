# JPYC-commerce-mcp 本格開発計画

最終更新: 2026-04-15

## プロジェクトの目的

JPYC（日本円ステーブルコイン）を使用して、エージェント同士が自律的に決済・交渉を行うMCPサーバー。取引履歴から信頼スコアを算出し、SBT（Soul Bound Token）として発行、その信頼スコアを次回以降の交渉材料として利用する「信頼ネットワーク」を構築する。

## コア価値

- エージェント間の**自律的な経済活動**（決済・交渉・契約）
- **信頼スコアの共有ネットワーク**により、初対面の相手とも安全に取引可能
- JPYC決済による円建てのシームレスな価値移転
- オンチェーン検証可能な信頼性の担保

## アーキテクチャ方針

### DB選定

**Neon（サーバーレスPostgres）の無料プランを採用**

理由:
- Supabase無料プランが枯渇しているため新規プロジェクト不可
- PostgreSQL互換なので既存Supabaseコードがほぼそのまま流用可能
- 無料枠で複数プロジェクト作成可能
- サーバーレスで自動スケール、スリープあり

**SQLiteは不採用**: MCPのローカル性質には合うが、各ユーザーのDBが分断され「信頼ネットワーク」の価値が失われるため。

### レジストリ構造

**中央共有レジストリ方式**を採用:

```
全ユーザーのMCP ──> [公式運営のNeon] (共有信頼レジストリ)
                    │
                    └─> Polygon (Merkle Rootで検証可能にする)
```

- 信頼スコアは全ユーザー共有
- Merkle Root の定期オンチェーンコミットで改ざん検出可能
- セルフホストも可能（環境変数で切り替え）

### 将来的なオンチェーン化

Phase 2以降、信頼スコアの確定値をSBTコントラクト経由でオンチェーンに書き、Neonはキャッシュ／インデクサに格下げ。真の source of truth をチェーンに移す。

## ライセンス・収益化戦略

### ライセンス

**MIT または Apache 2.0**（寛容ライセンス）でコア部分を公開。
- 採用されやすく、コミュニティが育ちやすい
- ネットワーク効果で公式レジストリが価値を持つ設計にして収益化

### 収益化モデル

**パターンA + パターンC の組み合わせ**

**Phase 1: フリーミアム SaaS**
- コードはMITで公開
- 公式信頼レジストリを有料ホスティング
  - 無料枠: 月間N件まで、read専用共有レジストリアクセス
  - 有料枠: 無制限、優先アクセス、SLA保証
- Sentry, PostHog型

**Phase 2以降: プロトコルフィー（Web3ネイティブ）**
- オンチェーンコントラクトに微額の使用料を組み込む
  - SBT発行時の手数料
  - 交渉成立時の決済に0.1%乗せる
  - 信頼スコア照会にmicropayment
- Uniswap, OpenSea型

### 守るべきポイント

- 秘密ロジック（アンチゲーミング判定、スコア調整パラメータ）はOSS対象外とし、サーバー側に保持
- CLA（Contributor License Agreement）でコントリビューション権利を集約
- JPYC・金融関連の規制リスクへの対応（ディスクレイマー明記）

## 技術スタック（確定分）

- **言語/ランタイム**: Node.js (ES Module) / JavaScript
- **プロトコル**: MCP (Model Context Protocol) via stdio
- **DB**: Neon (PostgreSQL互換、サーバーレス)
- **ブロックチェーン**: Polygon (JPYC決済)
- **決済トークン**: JPYC (ERC20)
- **信頼性検証**: Merkle Tree + オンチェーン Root コミット
- **将来的に**: ERC-5192 (SBT)
- **AI統合**: Claude API（タスク分析）

---

## 重要な設計思想

1. **ノンカストディアル**: MCPは秘密鍵を保有せず、トランザクション指示(calldata)を返すのみ
2. **オフチェーン処理 + オンチェーン検証**: 高頻度処理はDB、確定値のみMerkle Rootでチェーンにコミット
3. **ネットワーク効果の設計**: 信頼スコアは共有されるほど価値が上がる構造にする
4. **段階的Web3化**: まずSaaS的に運営 → ネットワーク確立後にオンチェーンへ移行

### ガバナンス方針

- **ノンカストディアル原則は絶対**：MCP はユーザーの秘密鍵を保有しない。calldata 返却のみ。例外は Merkle Root コミット用の運営ウォレットのみ（ユーザー資産とは完全分離）
- **公開ファイルでの禁止語**：`<禁止ワード1>`、`<禁止ワード2>` は使用禁止。「特許出願済」「patent-pending」は使用可（実際の禁止語句は非公開マスタ参照）
- **既存 GIFTERRA Supabase テーブルには一切触れない**：別プロジェクト、DB スキーマも完全分離
