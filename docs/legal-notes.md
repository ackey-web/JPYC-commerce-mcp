## 1. 非カストディアル原則とその根拠

### 設計上の事実

- **秘密鍵を一切保有しない**: MCPサーバーは署名鍵・ウォレット秘密鍵を保存・管理しない
- **calldata の返却のみ**: `execute_payment` ツールは `{to, data, value, gasEstimate, decoded}` を返すだけであり、トランザクションをブロードチェーンにブロードキャストしない
- **共有リレーウォレットなし**: サーバー側に共有リレーウォレットは存在しない
- **auto-approval の意味**: `request_human_approval` が `status='approved'` を返すことは「呼び出し元が自分のウォレットで実行してよい」という許可であり、MCPが代わりに実行するわけではない
- **送金主体は常に呼び出し元**: Polygon への署名・ガス支払いはすべて呼び出し元（ユーザー/エージェント）が担う

### 法的意義

この設計により、MCPサーバーは「資金の管理・移動を行う主体」ではなく「calldata 生成ツールと承認ゲートキーパー」として機能する。カストディアル規制（資金移動業・前払式支払手段発行業等）の直接的な対象にはなりにくい構造だが、以下の論点については正式確認が必要。

## 2. JPYC の資金決済法上の位置づけ

### 現状の分類候補

| 分類 | 根拠 | 判断に影響する要素 |
|---|---|---|
| **前払式支払手段（第三者型）** | 資金決済法 3 条 1 項、JPYC がサービス対価として利用可能 | 発行体（JPYC Inc.）の届出義務。MCP運営者は発行体でないため直接は非該当 |
| **資金移動業（第二種）** | 資金決済法 36 条、送金額が 100 万円以下の為替取引 | MCPが「為替取引を媒介」とみなされるかどうかが論点 |
| **ステーブルコイン（電子決済手段）** | 資金決済法 2 条 9 項（2023 年改正）、法定通貨担保型 | JPYC が同改正の「電子決済手段」に該当するかは JPYC Inc. の公式見解を要確認 |

### 論点メモ

- JPYC Inc. は現時点で前払式支払手段の届出事業者として運営しているとみられるが、2023 年資金決済法改正によるステーブルコイン規制との関係は流動的
- MCPサーバーが「JPYC の移動を指示する calldata を生成する」行為が「為替取引の媒介」に当たるかは、行為の実態・反復性・営利性等の総合判断となる
- **推奨**: Phase 1 前に JPYC Inc. のコンプライアンス担当および金融法専門弁護士に確認

## 3. レジストリ運営での手数料徴収と資金移動業該当性

### 想定シナリオ

Phase 2 以降でプロトコルフィー（例: JPYC 送金額の 0.1%）をレジストリ運営主体が徴収する場合の論点。

### 主要論点

**資金移動業への該当性**

- 資金決済法上、「為替取引」の媒介・代理を業として行うと資金移動業登録が必要
- MCPが「送金 calldata 生成 + フィー控除 calldata 生成」をセットで提供する場合、送金フローへの関与度が高まる
- 手数料をスマートコントラクトが自動控除する設計（コントラクトが媒介者）と、MCPサーバーが控除先を指定する設計では規制リスクが異なる

**前払式支払手段の「管理」**

- JPYC残高や信頼スコアを管理するレジストリが「前払式支払手段の管理業務」とみなされる可能性

**推奨設計方針（フィー設計時）**

1. フィー徴収ロジックはスマートコントラクトに閉じ込め、MCPはcontract addressを参照するだけにする
2. 運営主体は法人化し、必要に応じて資金移動業登録または電子決済手段等取引業の登録を検討
3. 手数料の性質を「コントラクト利用料（ガス代補填）」として設計し、「送金手数料」との混同を避ける

## 4. SBT 発行と有価証券該当性

### SBT（Soulbound Token）の特性

- 譲渡不可（ERC-5192 準拠）
- 信頼スコア・取引実績の証明として機能
- 金銭的リターンの期待や投資性はなし

### 金商法上の有価証券該当性分析

| 項目 | 判断 | 根拠 |
|---|---|---|
| 集団投資スキーム（ファンド）| **非該当** | 投資対象資産の共同出資・分配構造がない |
| 電子記録移転権利（いわゆる「ICOトークン」） | **非該当** | 譲渡不可なので市場流通・投資性がない |
| 信用情報として機能するスコアツール | **非該当** | 経済的権利を表章しない |

### 注意点

- SBT に将来的なガバナンス投票権・収益分配権を付与した場合は再評価が必要
- 「ポイント」類似の仕組みへの転用は前払式支払手段として別途整理が必要
- **特許出願済**の設計要素を含む場合、知財管理については別途確認

## 6. Phase 1 移行前の正式法務レビュー推奨事項

以下の事項について、金融規制専門の法律事務所によるレビューを Phase 1 前に実施することを強く推奨する。

### 必須確認事項

1. **JPYC の現在の法的分類確認**
   - JPYC Inc. の最新コンプライアンス状況（前払式 or 電子決済手段）
   - MCPが JPYC の移動に関与することによる運営者の義務

2. **MCPサーバー運営主体の登録義務確認**
   - calldata 生成・承認ゲートキーパーとしての機能が「資金移動業の媒介」に当たるか
   - レジストリ運営（信頼スコア管理・手数料徴収）の規制上の位置づけ

3. **フィー設計の事前審査**
   - プロトコルフィー（0.1%等）の徴収スキームが資金移動業登録を要するか
   - スマートコントラクト自動控除 vs. MCPサーバー指定の規制上の差異

4. **SBT 設計の確定前確認**
   - 将来的な機能拡張（ガバナンス権・報酬）を含む設計変更時の再評価タイミング

5. **利用規約・ディスクレイマーの法的妥当性確認**
   - README および公開ドキュメントのディスクレイマー文言の適切性
   - 責任制限条項の有効性（Apache 2.0 ライセンスとの整合性）

### ディスクレイマー文言案（README 掲載用）

## 7. 運営主体パターン比較

| パターン | メリット | リスク・懸念点 |
|---|---|---|
| **個人運営（フリーOSS）** | 立ち上げコスト最小 | 事業性認定時に個人責任、資金移動業登録は法人が原則 |
| **合同会社（LLC）設立** | 低コスト・柔軟、登録主体として機能 | 資本規制（資金移動業は最低純資産額あり） |
| **株式会社設立** | 信頼性・資本調達、金融機関対応 | 設立コスト・ガバナンスコスト |
| **既存法人（JPYC Inc. 連携）** | 既存ライセンス活用、法的障壁低 | 独立性・交渉コスト |
| **DAO/非法人** | 分散・コミュニティ運営 | 日本法上の権利義務帰属が不明確、規制対応困難 |

**Phase 0 推奨**: 個人運営のOSSとして公開しつつ、Phase 1 前に合同会社または株式会社設立を検討する。フィー徴収を伴う本番運用前に資金移動業の要否を確認。
