# GIFTERRA Commerce MCP — 実装指示（ARCHIVED）

## プロジェクト概要

GIFTERRAのコマースエージェントロジック（SBTによる信頼度評価・査定・交渉・送金・SBT更新）をMCPサーバーとして切り出す。

外部のAIエージェントがGIFTERRAの判断エンジンをツールとして呼び出せるようにすることが目的。

---

## デモシナリオ

「ScanTarotの機能追加タスクをGIFTERRA Commerce MCPが査定・交渉・送金まで完結させる」

フロー：
1. 発注エージェントがタスク要件を提示
2. MCPが受注エージェントのSBTプロファイルを読む
3. SBTスコア×タスク難易度で報酬額を算出
4. 交渉条件を提示
5. 人間が承認
6. JPYC送金実行
7. 完遂記録をSupabaseに書き込み、SBTランク更新

---

## 技術スタック

- MCPサーバー：Node.js + `@modelcontextprotocol/sdk`
- データ：GIFTERRAの既存Supabase（新テーブルを追加）
- 送金：JPYC on Polygon（デモフェーズはモック可）
- デプロイ：ローカルまたはRender（既存Node.jsサーバーに相乗り）

## Supabaseテーブル設計

既存のGIFTERRAテーブルには触らない。以下の4テーブルを新規追加する。

### mcp_agents
```sql
create table mcp_agents (
  id uuid primary key default gen_random_uuid(),
  wallet_address text unique not null,
  sbt_rank text default 'seed',        -- seed / broom / heaven
  contribution_score integer default 0,
  completion_count integer default 0,
  sentiment_score float default 0.0,
  created_at timestamp default now(),
  updated_at timestamp default now()
);
```

### mcp_tasks
```sql
create table mcp_tasks (
  id uuid primary key default gen_random_uuid(),
  description text not null,
  required_skills text[],
  deadline timestamp,
  difficulty_score float,
  recommended_reward_min integer,
  recommended_reward_max integer,
  status text default 'pending',       -- pending / negotiating / approved / completed
  created_at timestamp default now()
);
```

### mcp_negotiations
```sql
create table mcp_negotiations (
  id uuid primary key default gen_random_uuid(),
  task_id uuid references mcp_tasks(id),
  agent_wallet text,
  proposed_amount integer,
  rationale text,
  status text default 'pending',       -- pending / approved / rejected
  created_at timestamp default now()
);
```

### mcp_payments
```sql
create table mcp_payments (
  id uuid primary key default gen_random_uuid(),
  negotiation_id uuid references mcp_negotiations(id),
  from_wallet text,
  to_wallet text,
  amount integer,
  tx_hash text,
  task_result text,
  created_at timestamp default now()
);
```

## Tool定義と実装

### Tool 1: `get_sbt_profile`

**役割**：受注エージェントのSBT信頼度プロファイルを取得する

**入力**：
- `wallet_address` (string): 対象エージェントのウォレットアドレス

**出力**：
- `sbt_rank`: seed / broom / heaven
- `contribution_score`: 累積貢献スコア
- `completion_count`: タスク完遂数
- `sentiment_score`: センチメントスコア（0.0〜1.0）

**実装**：`mcp_agents`テーブルからwallet_addressで検索して返す。存在しない場合は初期値で新規レコードを作成して返す。

---

### Tool 2: `evaluate_task`

**役割**：タスク内容を査定して難易度と推奨報酬レンジを算出する

**入力**：
- `description` (string): タスクの説明
- `required_skills` (array): 必要スキルのリスト
- `deadline` (string): 締切日時（ISO形式）

**出力**：
- `task_id`: 生成されたタスクID
- `difficulty_score`: 難易度スコア（0.0〜1.0）
- `recommended_reward_min`: 推奨報酬下限（JPYC）
- `recommended_reward_max`: 推奨報酬上限（JPYC）

**実装ロジック**：
- required_skillsの数 × 重み係数で基礎スコアを算出
- deadlineまでの時間が短いほどスコア加算
- スコアに基づいて報酬レンジを計算（基準値：100 JPYC〜1000 JPYC）
- mcp_tasksテーブルに保存してtask_idを返す

---

### Tool 3: `propose_negotiation`

**役割**：SBTスコアとタスク評価を掛け合わせて交渉条件を生成する

**入力**：
- `task_id` (string): evaluate_taskで生成したタスクID
- `agent_wallet` (string): 受注エージェントのウォレットアドレス

**出力**：
- `negotiation_id`: 生成された交渉ID
- `proposed_amount`: 提示報酬額（JPYC）
- `rationale`: 算出根拠の説明文

**実装ロジック**：
- get_sbt_profileとevaluate_taskの結果を取得
- proposed_amount = recommended_reward_min + (recommended_reward_max - recommended_reward_min) × sentiment_score × sbt_rank_multiplier
- sbt_rank_multiplier: seed=1.0 / broom=1.2 / heaven=1.5
- rationaleはSBTランク・スコア・タスク難易度を含む日本語の説明文を生成
- mcp_negotiationsテーブルに保存

---

### Tool 4: `request_human_approval`

**役割**：人間に交渉条件の承認を求める（フロー内で唯一人間が介在するステップ）

**入力**：
- `negotiation_id` (string): propose_negotiationで生成した交渉ID

**出力**：
- `status`: approved / rejected
- `message`: 承認・否認の理由（任意）

**実装**：
- negotiation_idで交渉内容を取得して表示
- MCPのhuman-in-the-loopパターンで実装
- 承認された場合はmcp_negotiationsのstatusをapprovedに更新

---

### Tool 5: `execute_payment`

**役割**：承認された条件でJPYC送金を実行する

**入力**：
- `negotiation_id` (string): approved状態の交渉ID
- `from_wallet` (string): 送金元ウォレット
- `to_wallet` (string): 送金先ウォレット

**出力**：
- `payment_id`: 生成された送金記録ID
- `tx_hash`: トランザクションハッシュ（デモフェーズはモックハッシュ可）
- `amount`: 送金額

**実装**：
- negotiation_idのstatusがapprovedであることを確認
- デモフェーズ：tx_hashはモック文字列（`mock_tx_` + timestamp）
- 本番フェーズ：Polygon上でJPYC送金を実行
- mcp_paymentsテーブルに記録

---

### Tool 6: `update_sbt_record`

**役割**：タスク完遂後にSBTレコードを更新してランクアップを判定する

**入力**：
- `payment_id` (string): execute_paymentで生成した送金記録ID
- `task_result` (string): 成果物の説明または評価

**出力**：
- `updated_score`: 更新後の貢献スコア
- `new_rank`: 更新後のSBTランク
- `rank_up`: ランクアップしたかどうか（boolean）

**実装ロジック**：
- completion_countを+1
- contribution_scoreにamountを加算
- ランク判定：
  - seed → broom: contribution_score >= 600
  - broom → heaven: contribution_score >= 6000
- mcp_agentsテーブルを更新
- mcp_paymentsにtask_resultを記録

## 実装の優先順位

1. Supabaseテーブル4つを作成
2. MCPサーバーの骨格（index.js）を作成
3. get_sbt_profile → evaluate_task → propose_negotiation の順に実装
4. request_human_approval を実装
5. execute_payment をモックで実装
6. update_sbt_record を実装
7. デモシナリオを一通り流して動作確認

## 注意事項

- 既存GIFTERRAのテーブルやロジックには一切触れない
